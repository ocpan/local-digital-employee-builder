# 规模化推广手册：帮多人/多部门复制数字员工

> 场景：你已经跑通了自己的分身，现在要帮同事、其他业务中心各搭一个。
> 核心认知：这是**复制 + 实例化**问题，不是重新发明。底座只有一套，差异全部收敛进配置、知识库、人格三处。

## 0. 先判断：这个对象值不值得搭

不是每个人都适合搭数字员工。先过四道闸（任一不满足就先补，别硬上）：
1. **有稳定知识源吗**？有可沉淀成仓库的文档/纪要/规范 → 能做 RAG 问答；没有 → 只能做纯工具型，价值窗口窄。
2. **高频任务能自动化吗**？跑一遍 [[ai-agent-product-research]] 的 Agent 化判断。全是需要人拍板的判断题 → 不适合。
3. **要连的业务系统有 CLI/API 吗**？要查 某业务系统/工时/审批等系统 → 得有可编程入口（像 业务系统CLI）；纯 UI 系统只能靠人。
4. **飞书应用能发布吗**？企业自建应用要过管理员审核、配可用范围、开消息权限——组织卡这一步就上不了线。

## 1. 每个分身的独立要素（别共用）

| 要素 | 为什么必须独立 |
|---|---|
| 飞书自建应用（app_id + secret） | 一个 app 只能有一个事件消费者；共用会互相抢事件流 |
| 常驻服务（launchd/systemd label） | 各自的进程、日志、重启策略 |
| 知识库仓库 | 每人的知识边界、访客可见范围不同 |
| 业务系统凭证目录 | 各自的账号/权限，隔离到独立 HOME/profile，避免互相覆盖 |
| PROTOCOL 人格文件 | 名字/owner/业务/分权口径都不同 |
| config.json | 所有实例差异的收敛点 |

**可共用**：runner 代码（一份，靠 config 驱动）、大脑后端（同一个 LLM 网关，各自 brain_home）、卡片/标记协议、进阶能力模块。

## 2. 实例化流程（给一个新对象搭的标准步骤）

1. **预研**（[[ai-agent-product-research]] 的数字员工快速通道）：产出这个对象的高频任务、要连的系统、访客策略、要不要定时产出、成功指标。
2. **建知识库**：把对方的文档沉淀成可检索仓库（类别/业务线/产品 三级 + rag_search 脚本起步）。缺数据就先补数据，别急着搭对话。
3. **复制底座**：拷 runner 骨架，改 `config.json`——机器人名、owner open_id/chat_id、知识库路径、大脑 provider、业务系统 bin/凭证、访问策略（open/whitelist）、卡片文案、定时任务条目。
4. **写 PROTOCOL**：套 [assets/PROTOCOL.template.md](../assets/PROTOCOL.template.md)，替换身份/知识源/业务/分权口径 + 从 [advanced-capabilities.md](advanced-capabilities.md) 挑需要的协议段（长任务、授权自愈话术、自我改造安全）。
5. **接业务系统**（如需）：按标记协议家族接 `[[X查询|...]]` 只读通道 + 白名单闸门；凭证走 SSO 自愈方案。
6. **飞书上线**：申请 scope（im:message / im:message:readonly / contact:contact.base:readonly）→ 发版 → 管理员审核 → 配可用范围 → 开群消息权限。
7. **实测回放**：真实账号发 3-5 条高频问题，验证路由、分权、来源引用、卡片渲染。
8. **按痛点上进阶能力**：先"提速稳定"（预检索+瘦身+抖动重试），再"后台执行器"地基，再定时/写作类。

## 3. 交付给非技术对象时

对方（同事/业务中心负责人）通常不懂技术，交付要包含：
- **一句话能力边界**："它能答 X 知识库、查 Y 系统只读数据、每天早上推 Z 报告；不能改数据、不能替你决策。"
- **给谁用**：本人全权限 vs 访客只读，写清楚。
- **怎么触发**：日常问答 / `/tasks` / 定时产出，各给一个例子。
- **失效自救**：授权过期怎么办（桌面重登脚本）、机器人不回复找谁。
- **数据与隐私说明**：读了哪些数据源、访客看不到什么——这条最容易被追问。

## 4. 常见反模式（血泪）

- **无脑套现有分身的 config**：别人的高频任务、业务系统、访客策略可能完全不同。必须先预研。
- **多个分身共用一个飞书 app**：事件流互抢，都收不到消息。一人一 app。
- **一次把 12 个进阶能力全上**：对方可能只需要"知识问答 + 每日晨报"。按痛点排，别镀金。
- **知识库没就绪就搭对话**：没数据源的问答分身只会编，不如不上。
- **把凭证共用**：业务系统 token 互相覆盖，隔离到独立目录/profile。
- **跳过飞书发布审核估工**：组织审核可能要几天，排期务必留出。
- **不做实测就交付**：卡片渲染、分权、来源引用只有真实回放才暴露问题。

## 5. 复制成本参考

底座已被复制出"某部门数字员工（示例）"（同套 runner + 三档权限 + 某业务系统 数据层），证明可复制。单个新分身的主要成本不在代码（底座现成），而在：知识库整理（数天，取决于资料量）、业务系统接入（有 CLI 则 1-2 天）、飞书发布审核（组织流程，不可控）。技术工作量通常小于这三项。
