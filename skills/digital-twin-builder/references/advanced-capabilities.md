# 进阶能力手册（从 示例Agent 2026-07 迭代提炼）

> 基础五层架构跑通后的第二阶段能力。每节 = 一个可独立移植的模式：解决什么问题 → 怎么做 → 坑位。
> 参考实现：`your-digital-twin/runner/runner.py`（单文件内按段组织，搜索段落标题即可定位）。

## 1. 标记协议家族（大脑↔runner 的动作总线）

**问题**：大脑跑在沙箱里（访客 read-only 无网络），但要查外部系统、起后台任务、转交事项。
**模式**：大脑在回答文本里输出结构化标记，runner 在沙箱外解析执行、结果回喂，循环 N 轮（设上限防死循环）：

| 标记 | 作用 | 权限 |
|---|---|---|
| `[[某业务系统查询\|环境\|参数]]` | 外部系统只读查询，runner 校验后代跑 | 所有人 |
| `[[某业务系统认证\|环境\|cookie]]` | 凭证兜底通道 | 仅本人主动提供 |
| `[[后台任务\|自包含描述]]` | 转后台异步执行 | 仅本人（runner 强制） |
| `[[任务完成]]总结[[/任务完成]]` / `[[任务失败\|原因]]` | 后台任务收尾 | worker 轮内 |
| `[[转交<owner>]]...[[/转交]]` | 需求/事项转交本人 | 访客场景 |
| `[[图片:路径]]` | 上传换 img_key 嵌卡片 | 所有人 |

**必配防泄漏兜底**：回复发出前，凡是残留**未闭合**的 `[[` 标记前缀（轮次耗尽/输出被截断），整段截掉；截空才给兜底话术（注意：空回复可能是合法路径——比如只输出了转交块——不能无条件兜底）。

## 2. 后台任务执行器（突破单轮超时）

**问题**：headless 大脑单轮有硬超时（如 480s），盘点整库/批量流转这种小时级任务干不了。
**模式**：队列 + 串行 worker + 多轮 resume 续跑。
- 入队：大脑输出 `[[后台任务|描述]]`（描述必须**自包含**——后台无对话上下文）→ 持久化队列（JSON 文件够用）+ 发一张专属进度卡（记 message_id）。
- 执行：worker 每轮给大脑独立超时预算（如 1500s），最多 N 轮（如 12），轮间用**同一会话 resume**（上下文不丢）；轮内支持标记协议回路；单轮超时不算失败、续下一轮。
- 收尾：`[[任务完成]]`/`[[任务失败]]` 标记；进度卡 PATCH 成最终报告 + **另发一条轻提醒**（旧卡会被刷远）。
- 管理：`/tasks` 列表、`/task cancel <id>`（轮次边界生效即可，别追求杀进程）。
- 容灾：先登记再处理（异常不重复）；重启时 running→queued 复位（会话可 resume 所以能续）。
- 定时任务、写作流水线等上层能力都骑在这个队列上，**先做它**。

## 3. 通用定时任务调度器（主动性地基）

**模式**：config 里声明式配置多条目，调度线程每分钟扫一遍，到点的**投递到后台任务队列**（自己不执行）：
```json
"scheduled_tasks": {
  "daily_briefing":     {"enabled": true, "label": "每日晨报", "workdays_only": true, "time": "09:00",
                          "grace_minutes": 15, "desc": "<自包含任务描述>"},
  "weekly_work_summary": {"enabled": true, "label": "每周工作汇总", "weekday": 4, "time": "17:00", "desc": "..."}
}
```
要点：`weekday` 缺省=每天、`workdays_only` 跳周末、`grace_minutes` 宽限窗（错过点了还能补）、**按 last_run_date 防同日重复**、时区用 ZoneInfo。晨报/周报这类"信息流汇总"任务的产出可直接放任务完成卡，不必建文档。

## 4. 外部系统授权自愈（绝不向用户索要凭证）

**反模式**：凭证失效时让用户在对话里贴 Cookie（示例Agent 早期做法）——体验差、泄露风险、群聊灾难。
**模式**：
1. 凭证用 SSO Token（自动刷新），CLI 级 `auth login` 一次性授权；
2. runner 保活线程定期（20min）跑 `auth status` 触发主动续期；
3. 失效检测要**活体判别分档**：token 死（→告警 owner + 用户只看到"已通知处理"）vs token 活但个别旧接口仍要 Cookie（→单项跳过不打断整轮）；
4. 告警按环境+类型节流（10min），附**桌面一键重登脚本**（mac `.command` / win `.cmd`，只登失效环境、登完自检、runner 免重启）；
5. PROTOCOL 明写：遇 `AUTH_EXPIRED` 只告知用户稍后再试，**不要向任何人索要凭证**。

## 5. 只读安全闸门（白名单默认拒绝）

**反模式**：用"写动词黑名单正则"拦写命令（save|create|update…）——`view update` 这类会被"view 像只读"误放，新命令直接漏。
**模式**：按 CLI 的 **top command + subcommand 精确白名单**，不在名单内一律拒绝；再叠一份高危命令黑名单做双保险。拒绝话术要列出可用的只读命令引导大脑换写法。

## 6. 大脑健壮性（哨兵 + 重试 + 可诊断）

- **必须捕获流式事件里的 `error` / `turn.failed`**（codex 的失败原因在 stdout 事件流、不在 stderr——不接的话失败日志永远是空的，排障靠猜）。
- 失败分类哨兵：`BRAIN_TIMEOUT`（超时→提示用户"分批执行/转后台"）、`BRAIN_BACKEND_ERR`（企业网关抖动：response.failed/流断/5xx → **不丢会话**、5s 后自动 resume 重试一次，二连败才报错并提示"稍后重发"）。普通失败才用通用报错。
- 企业 LLM 网关（如 企业 LLM 网关）会抖动：小请求靠 SDK 自动重连能活、大请求重连额度耗尽必挂——重试机制是刚需不是锦上添花。

## 7. 提速三板斧

1. **runner 侧知识库预检索**：知识类问题由 runner 先跑 rag_search 把结果注入提问，省掉大脑"决定检索→跑命令→读结果"整轮往返（实时数据类问题要识别跳过）。**中文分词坑**：朴素索引对连续中文按 2-8 字贪婪切块，整句问句会切成无意义碎片→0 命中；查询侧先用停用词把问句剁成短语；且"是否值得预检索"的判定必须用保留问句词的原文（停用词剁掉"怎么/如何"会误判）。
2. **系统提示瘦身**：先测量（技能清单可占系统提示 50%+），再动手——排除对话用不到的开发期 skill + 描述截断（触发语通常在前 110 字符）。
3. reasoning 档位：对话型分身调低 reasoning 档位提速（上线后用你自己的中位耗时做前后对比）。

## 8. DM 代答 / 私聊代理（含教训）

对齐飞书原生"私聊代理"的自建增强版（原生不接知识库/外部系统）。开关集：触发方式（smart 仅领域相关/all 任何真问题）、回复风格预设（mimic/natural/concise/detailed）、每人冷却、每小时上限、署名开关、自定义铁律指令。
**关键设计**：
- 召回用**全量近窗消息 + "像提问"廉价预过滤**（问句特征词/领域词兜底），不要用关键词搜索召回——领域词列表永远堵不完（真实漏抓案例：一句“某某流程怎么加人”不含任何预设关键词，就被关键词召回漏掉）。
- 回复通道选**拉群**（提问人+本人+bot 三方群，按提问人复用群），不要 bot 私聊陌生人；本人账号只发一句标注"自动消息"的引导。
- **致命教训**：`人工优先窗`（本人 N 分钟内回过就不代答）依赖消息搜索接口 → 搜索有**分钟级索引延迟**，本人刚回复也可能检测不到 → 误代答。轮询感知天然不可靠，这功能要真稳，需要实时事件（用户消息事件/回调）；没有就别默认开。

## 9. 观测与自愈

- `/stats [天数]`：直接从 runner 日志正则统计（提问量分身份/成功率/耗时中位 P90/各通道量），不用额外埋点。
- **心跳**：定期查事件总线 consumers 数，=0 说明"进程活着但聋了"（bot API 独立于事件流所以还能发告警卡）；彻底宕机靠 launchd/systemd KeepAlive。
- **测试日志必须隔离**：单测 load 真模块时把 LOG_FILE 重定向到临时文件，否则测试日志污染运行统计。
- 运行基线：上线后统计提问量/成功率/中位耗时/P90/访客占比，作为后续优化的对标基线。

## 10. 自我改造安全规则（bot 改自己的代码）

owner full-access 沙箱下，分身可能被要求（或自发）修改自身 runner/config——示例Agent 真实发生过：为实现"每周定时周报"自己写了调度器、跑了测试、重启了自己，**但重启杀掉了自己正在回复的轮次，用户没收到任何汇报**。PROTOCOL 必须内置：
1. 仅 owner 明确确认后可改自身代码/配置；
2. 改完必须跑测试**全绿**才生效，失败回滚并汇报；
3. **先把"改了什么+测试结果"的汇报发出去，再重启**；
4. 重启优先建议人工执行；自动重启前发"即将重启"卡片。

## 11. 长任务协议（PROTOCOL 模板增补）

系统提示里明写两条路径：本人任务优先 `[[后台任务]]`（方式一）；访客/需逐步确认的交互式任务走"拆批+每轮一批"（方式二）。超时报错话术直接教用户说"分批执行/转后台"。

## 12. 对外称谓与渠道原生功能

- 对外自称统一定一个词并写进 PROTOCOL（示例Agent 用「AI 助手」，弃用「数字分身」——用户反馈后全局改）。
- 预研时先查渠道**原生能力**：飞书已有原生"私聊代理/数字分身"（触发方式/回复风格/冷却/上限/署名/自定义指令）——自建版的差异化必须是"接本人知识库+业务系统"，否则别自建。
