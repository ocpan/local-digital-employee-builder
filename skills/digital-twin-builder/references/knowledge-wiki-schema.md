# 知识库进阶：LLM Wiki 编译层搭建规范

> 本文档是知识库的"进阶配方"。基础版（docs + rag_search 关键词检索）足以起步；
> 当你的知识库文档量增长、原文散碎难检索时，加这层"编译层"能显著提升检索质量——
> 让 LLM 帮你把散落原文综合成结构化百科词条，检索时优先命中编译产物。

---

## 一、三层知识模型概览

在基础的"丢文档 + 切块检索"之上，叠加一个由 LLM 维护的"编译层"：

| 层 | 目录 | 谁维护 | 可变性 | 是否进 RAG |
|---|---|---|---|---|
| **第一层 · 原始资料** | `docs/` | 飞书同步 / 归档脚本 | LLM **只读，不改写** | ✅ layer=raw |
| **第二层 · 编译产物** | `wiki/` | **LLM 按本规范生成/维护** | 可重写 | ✅ layer=wiki（加权） |
| **第三层 · 规则** | `wiki/purpose.md` / `wiki/schema.md` | 人 + LLM | 低频更新 | ❌ 导航用 |

> 核心原则：**人负责选材料、问问题、做判断；LLM 负责总结、归档、交叉引用、维护一致性**。
> 原文永不被改写，所有综合结论都落在 `wiki/`，并通过 `sources` 回链原文——保证可追溯。

---

## 二、目录约定

```
knowledge_base/
├── docs/                   # 原始资料（只读权威来源）
│   ├── 产品/               # 按业务线/产品分目录
│   └── 日常工作/           # 方法论/经验等
├── wiki/                   # LLM 编译产物
│   ├── purpose.md          # 方向锚（知识库目标与边界）
│   ├── overview.md         # 全局综合摘要
│   ├── schema.md           # 编译规范（本文件的落地版）
│   ├── review-queue.md     # 矛盾/待审核项
│   ├── entities/           # 实体页：产品/系统/模块/接口/字段
│   │   └── {实体名}.md
│   └── concepts/           # 概念页：方法论/规则/机制
│       └── {概念名}.md
├── rag/                    # 检索索引（脚本生成）
│   ├── index/              # documents.jsonl + keywords.json
│   └── chunks/             # 切片文件
└── scripts/
    ├── build_rag_index.py  # 重建索引（同时扫 docs + wiki）
    └── rag_search.py       # 关键词检索
```

- `entities/` 与 `concepts/` 下的页面会被索引为 `layer=wiki` 切片，检索时**优先于 docs 命中**。
- 导航类文件（purpose / overview / schema / review-queue）不进 RAG，由 Agent 直接整篇读取。

---

## 三、页面 frontmatter 规范

所有 `wiki/entities/` 和 `wiki/concepts/` 页面必须带 YAML frontmatter：

```yaml
---
title: 页面标题
type: entity | concept
business_line: 业务线名（与 docs/ 目录对齐）
product: 产品名（无明确产品写"通用资料"）
updated_at: YYYY-MM-DD
sources:                       # 回链贡献本页内容的原始文档
  - docs/产品/某系统/某需求.md
  - https://your-wiki-url/...
related: [关联实体名, 关联概念名]
status: active | draft | needs-review
---
```

- `business_line` / `product`：决定 RAG 切片归属，必须与 `docs/` 分类口径一致。
- `sources`：**可追溯性的核心**，每条结论都应回链到原文。
- `related`：正文中用 `[[页面名]]` 交叉引用的对象。

---

## 四、编译流程（Ingest 两步思维链）

向 wiki 写入时遵循"先分析、后生成"，不要跳过分析直接写：

### 第一步 · 分析

1. 先读 `wiki/purpose.md`（方向锚）、`wiki/schema.md`（本规范）、目标产品已有 wiki 页。
2. 读取待摄入的原始文档（`docs/...`）。
3. 输出结构化分析报告：
   - 关键实体（产品/系统/模块/接口/字段）
   - 关键概念（方法论/规则/机制）
   - 主要论点与结论
   - 与现有 wiki 的关联（应更新哪些已有页面）
   - 矛盾与张力（与既有结论冲突处 → 进 review-queue）
   - 结构建议（新建或更新哪些页面）

### 第二步 · 生成

1. 按分析报告新建/更新 `entities/`、`concepts/` 页面，遵守 frontmatter 规范。
2. 更新受影响页面的 `sources`、`related` 与正文 `[[link]]`。
3. 增量更新 `wiki/overview.md`（新增产品、规模变化）。
4. 把矛盾/重复/断链问题写入 `wiki/review-queue.md`。

---

## 五、页面写作要求

**实体页结构建议**：定位 → 业务背景/痛点 → 核心能力/模块 → 关键规则与机制 → 对接系统 → 跨类别原文索引 → 待确认。

**概念页结构建议**：定义 → 适用场景 → 方法/规则 → 示例 → 关联实体 → 来源。

**通用规则**：
- 只写"网上搜不到"的专有结论；通用常识不沉淀。
- 不臆测：原文未覆盖的，标注"待确认"，不编造。
- 一篇原始文档常同时影响多个 wiki 页面——这正是编译层的价值，不要回避重复链接。
- 跨类别打通是重点：同一产品的需求/设计/配置/操作应在其实体页用 link 串起来。

---

## 六、交叉引用（wikilinks）

- 引用其他 wiki 页面：`[[页面名]]`（页面名 = 文件名去扩展名）。
- 引用原文：markdown 链接 + doc_path，如 `见[某需求](docs/产品/某系统/xxx.md)`。
- 跨页面交叉引用是这套模型的核心价值——把散落的信息编织成网。

---

## 七、重建索引

修改 wiki 内容页后，需重建 RAG 索引：

```bash
python3 scripts/build_rag_index.py
```

索引器同时扫描 `docs/`（layer=raw）与 `wiki/entities`、`wiki/concepts`（layer=wiki）。
wiki 层标记为 `layer=wiki` 后，检索时加权优先返回——这使得编译产物比冗长原文更容易被命中。

---

## 八、方向锚（purpose.md）模板

每个知识库应有一份 `wiki/purpose.md` 回答：

1. **服务谁**：这个知识库给谁用、在什么场景下调用。
2. **解决什么**：库存在的理由（核心问题列表）。
3. **方法论立场**：默认遵循的原则（如"配置优先""不臆测"等）。
4. **范围与边界**：收纳什么 / 不收纳什么。
5. **当前生长方向**：主动补全的优先项。

它是整个知识库的"方向盘"——LLM 每次摄入和检索前先读它，确保不偏移。

---

## 九、何时启用编译层

**不是所有知识库都需要这一层**。判断标准：

| 情况 | 建议 |
|------|------|
| 文档 < 20 篇，结构清晰 | 基础 docs + rag 足够，不加 wiki 层 |
| 文档 > 50 篇，多产品/多版本，信息碎片化 | ✅ 加 wiki 编译层，收益明显 |
| 同一主题散落在 5+ 份文档里 | ✅ 编译成一个实体页，串联可追溯 |
| 需要频繁回答"跨文档综合问题" | ✅ wiki 概念页就是为这类问题设计的 |

**启用步骤**：
1. 创建 `wiki/` 目录 + `purpose.md` + `schema.md`（可直接复制本文件）。
2. 让 AI 工具读 schema，对已有 docs 跑一轮"分析+生成"。
3. 重建索引，验证检索质量提升。
