# 方法论动作清单（从真实项目实录蒸馏）

> 原始素材是一次"从零把一个平台做出来"的完整实操记录。对外版去掉了具体项目细节，
> 只保留**可复用的动作标签**——这些标签就是方法论本身。搭你自己的项目时，按这套顺序推进。

## 阶段 A · 资料收集（全景先行）

- 全景探索：先读整个 workspace/仓库目录结构，建立全局认知，再动手。
- 现有系统分析：分析要对接系统的代码结构（Controller/Service/DAO 等）与接口。
- 文档收集与转换：把需求/设计文档统一抓取、转 Markdown 归档。
- 数据对齐：把不同来源（文档、链接、代码）做精确映射，消除歧义。
- 参考分析：分析已有 Demo/原型，提取可复用的页面结构与交互。
- 元数据管理：对"负责人/修改时间/创建时间"等做结构化提取。
- 知识库入库 + 完整性校验：内容入库后校验完整，不遗漏。
- 批量入库：文档多时用工具批量抓取正文，别手抄。

## 阶段 B · 需求文档与概要设计

- 需求文档编写：按模块拆分成产品需求说明书。
- 冲突检查：交叉比对不同来源文档的信息冲突。
- 概设编写：写实现蓝图（implementation-blueprint）。
- 技术评估：写后端对接可行性分析、鉴权机制评估。
- 标识预定义：把要对接系统的配置标识（空间/工作项/属性/选项）清单先定义好。
- 接口设计：产出 BFF/接口契约文档。
- 部署设计：补 Docker/部署架构。
- 优先级决策：用 KANO 等做需求优先级排序，写 P0 交付计划。

## 阶段 C · Demo 原型搭建

- Demo 框架：优先单文件 SPA 快速验证交互路径。
- 页面构建：逐页面对照参考构建。
- 全盘盘点：定期全量检查裂图、重叠、错位、样式问题。
- 世界级打磨：按"世界顶级同类产品"的标准做视觉（毛玻璃、微光、动画过渡等）。
- 架构优化：Demo 长大后拆多页面、抽公共组件（shared.css/shared.js）。

## 阶段 D · BFF 与接口对接

- BFF 搭建：初始化后端项目（如 FastAPI + pyproject）。
- 环境配置：定义 Docker Compose 开发环境。
- 接口对接：BFF 路由 + 目标系统对接适配层。

## 阶段 E · 本地环境与持续打磨

- 本地验证：尽量不连 VPN 先在本地跑通，缩短反馈环。
- 截图驱动：对照真实截图逐页重构/优化。
- 素材提取：从参考 Demo/设计稿提取真实图片素材替换占位。
- 差异审计：输出 UI 风格差异报告，逐项对齐。
- 全盘盘点（再次）：主动找遗漏模块补齐，而不是等人提。

## 贯穿全程的方法论原则（原话精神）

- "把它当作一个完整的项目来做"——不是拼个demo应付，按完整项目的标准推进。
- "用世界顶级同类产品的设计思维来做"——视觉与体验对标行业最高水准。
- "全盘盘点裂图、重叠、错位问题"——定期主动全量自检，别靠别人发现。
- "先试本地能不能在不连 VPN 的情况下跑通"——优先缩短验证链路。
