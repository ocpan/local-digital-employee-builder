# 本人 产品搭建方法论

> 从一个复杂 B 端平台（示例）的完整搭建过程中提炼的产品方法论，覆盖从资料收集到可交互 Demo 的全流程。

## 核心理念

1. **文档先行，代码跟进** — 先把所有需求、概设、接口契约整理成结构化文档，再动手写代码
2. **复用优先，增量创新** — 优先分析已有系统（如 某业务系统）的能力边界，能复用的复用，需要新建的再新建
3. **世界顶级标准** — 每一步都以世界顶级平台的标准来要求，从 UI 到交互到架构
4. **迭代验证，逐步推进** — 不求一步到位，但每一步要可验证、可演示

## 适用场景

- 从零搭建一个新平台/产品
- 基于已有系统做衍生平台
- 需要对接低代码/配置化平台（如 某业务系统）
- 需要快速产出高保真可交互 Demo

## 第一阶段：资料收集与知识库搭建

### 目标
把散落在各处的需求文档、设计文档、概设文档统一归档，形成结构化知识库。

### 步骤

1. **收集所有相关文档来源**
   - 飞书 wiki / 云文档链接
   - 本地已有的 Word/PDF 文档
   - 已有代码仓库中的 README / 设计文档
   - 设计稿及切片导出（如 Figma / Sketch 等设计工具）

2. **统一转换为 Markdown**
   - 所有文档转为 `.md` 格式
   - 按类别归档到 `docs/` 目录
   - 分类方式：业务需求 / 概设文档 / 接口文档 / 对接文档
   - 保持完整正文，尤其是表格内容要做格式适配

3. **补全与校验**
   - 逐篇检查内容是否完整，有无缺漏
   - 表格格式是否可识别
   - 与原始文档对比，确保信息未丢失
   - 多来源文档做冲突检查与合并

### 关键动作
```
收集来源 → 批量转 MD → 按类别归档 → 完整性校验 → 冲突合并
```

### 经验教训
- 飞书 wiki 页面动态渲染，直接抓取可能内容不全，优先用 MCP 或导出功能
- docx 内嵌链接映射要用 hyperlink r:id 做精确匹配，不能按 rels 顺序绑定
- 先建目录清单，再逐篇补正文，比一次性全量导入更稳健

## 第二阶段：现有系统分析与复用评估

### 目标
全盘分析已有系统的代码、架构、API，评估复用可行性。

### 步骤

1. **代码全景探索**
   - 遍历项目目录结构（前后端、配置、构建脚本）
   - 识别技术栈：语言 / 框架 / 数据库 / 构建工具
   - 找到关键入口点：pom.xml, package.json, Dockerfile

2. **后端能力评估**
   - 梳理所有 API 端点（Controller → Service → DAO）
   - 识别数据模型（Entity / DTO / VO）
   - 分析鉴权/权限机制
   - 评估 Open API 的复用潜力

3. **前端代码评估**
   - 分析路由结构、页面组件
   - 识别可复用的 UI 组件和交互模式
   - 评估 API 调用方式，是否可对接

4. **输出复用方案**
   - 哪些模块可直接复用
   - 哪些需要适配层（如 BFF）
   - 哪些必须新建
   - 接口对接策略

### 关键动作
```
代码探索 → 技术栈识别 → API 梳理 → 数据模型分析 → 复用方案输出
```

### 架构决策模式
- **复用后端 + BFF 适配层** — 适用于低代码平台（某业务系统），通过 BFF 隔离字段差异与鉴权复杂度
- **空间标识 / 工作项标识 / 属性标识** — 对接配置化平台时预定义好所有标识编码，到生产环境直接配置
- **权限复用** — 利用已有平台的权限管理模块，设定用户组和数据访问权限

## 第三阶段：产品需求与概要设计

### 目标
产出完整的产品需求说明书和概要设计文档。

### 步骤

1. **需求文档编写**
   - 参照已有系统的需求文档格式
   - 按模块拆分，定义每个模块的功能边界
   - 包含：功能描述 / 用户角色 / 操作流程 / 数据关系

2. **定义数据契约**
   - 工作项类型定义（如：项目计划、Top Issue、评审日历、交付物管理、人员配置）
   - 属性标识清单
   - 接口标识与参数规范
   - 选项标识预定义

3. **概要设计文档**
   - 整体架构图
   - 模块间依赖关系
   - 对接方案（与已有系统的接口契约）
   - 权限设计
   - 部署架构

4. **标识清单输出**
   - 输出 MD 格式的标识清单文档
   - 包含空间标识、工作项标识、用户组定义
   - 确保到生产环境可直接配置

### 关键动作
```
需求编写 → 数据契约定义 → 概设文档 → 标识清单输出
```

## 第四阶段：Demo 原型搭建

### 目标
产出一个可本地运行、可交互的高保真 Demo，用于汇报和需求验证。

### 步骤

1. **建立基础框架**
   - 单文件 SPA（HTML + CSS + JS），轻量级无依赖
   - 16:9 视口容器模拟真实应用
   - Hash-based 路由实现页面切换
   - 侧边导航栏 + 顶栏 + 内容区布局

2. **参考现有设计**
   - 分析设计稿 / POC 原型
   - 提取关键 UI 模式和交互行为
   - 明确每个页面的布局结构

3. **世界级 UI 标准**
   - 毛玻璃质感、微光效果、圆角卡片
   - 图片 crossfade 过渡动画
   - 悬浮放大、边框高亮等交互反馈
   - 深色主题 + 品牌色点缀（按产品品牌色取值）
   - 自定义滚动条、backdrop-filter

4. **设计稿转代码工作流**
   - 设计稿切片图按功能/页面分目录存放
   - 命名规范：`D{编号}_{场景名}_{中文名}.png`
   - 按模块逐页构建：先骨架布局，后填充图片和交互
   - 图片路径保持与切片目录一致

5. **迭代打磨**
   - 逐页检查：图片是否对应、交互是否正确、样式是否统一
   - 全盘盘点裂图、重叠、错位问题
   - 浏览器实时验证每次修改

### 关键动作
```
基础框架 → 设计稿分析 → 骨架布局 → 图片填充 → 交互绑定 → 世界级打磨 → 迭代验证
```

### 页面构建模式（以列表页 / 详情页 / 配置页为例）
```
顶栏 Tab → 工具栏 / 筛选器 → 左面板（下拉 + 搜索 + 网格卡片） → 主内容区（hero 区域） → 底部工具栏 → AI 浮动按钮
```

## 第五阶段：本地环境验证

### 目标
确保代码能在本地跑通，验证对接可行性。

### 步骤

1. **工具链安装**
   - 确认所需工具：Java/Maven/Node.js/Python 等
   - 安装缺失工具（如 Maven、特定 JDK 版本）

2. **后端本地启动**
   - 分析配置文件（application.yml / profiles）
   - 处理数据库/中间件依赖
   - 尝试本地启动（可能需要 VPN 或 mock）

3. **前端本地启动**
   - 静态文件服务：`npx serve` 或 `python -m http.server`
   - 确保图片路径正确
   - 验证交互功能

4. **对接验证**
   - 前端调后端 API 是否通
   - 数据格式是否匹配
   - 权限是否正确传递

### 关键动作
```
工具安装 → 后端启动 → 前端启动 → 对接验证
```

## 方法论总览

```
阶段一：资料收集与知识库搭建
  ↓
阶段二：现有系统分析与复用评估
  ↓
阶段三：产品需求与概要设计
  ↓
阶段四：Demo 原型搭建
  ↓
阶段五：本地环境验证
  ↓
（循环迭代）
```

## 核心原则清单

| # | 原则 | 说明 |
|---|------|------|
| 1 | 文档先行 | 不管多急，先把需求和设计文档整理清楚 |
| 2 | 复用优先 | 先看已有系统能给什么，再决定自己做什么 |
| 3 | 世界顶级标准 | 每一个 UI 细节、每一个交互都对标世界级产品 |
| 4 | 标识预定义 | 对接配置化平台时，提前定义好所有标识编码 |
| 5 | 迭代验证 | 每做一步就验证一步，浏览器打开看效果 |
| 6 | 全盘盘点 | 定期做全局检查，发现重叠、错位、遗漏 |
| 7 | 设计稿驱动 | 以设计稿为准绳，不是自由发挥 |
| 8 | 一步步来 | 明确步骤，按步骤执行，不跳步 |
