# 产品搭建方法论（底座）

> 被 digital-twin-builder 和 ai-agent-product-research 两个 skill 引用的通用底座。
> 适用于从零搭建一个新产品/平台/数字员工的全过程。

## 核心理念

1. **文档先行，代码跟进** — 先把需求、设计、接口契约整理成结构化文档，再动手写代码
2. **复用优先，增量创新** — 优先分析已有系统的能力边界，能复用的复用，需新建的再新建
3. **迭代验证，逐步推进** — 不求一步到位，但每一步要可验证、可演示
4. **全盘盘点** — 定期做全局检查，发现遗漏和冲突

## 适用场景

- 从零搭建一个数字员工 / AI Agent
- 基于已有系统做衍生产品
- 需要快速产出可工作的 MVP

---

## 阶段一：资料收集与知识库搭建

### 目标
把散落在各处的需求文档、设计文档统一归档，形成结构化知识库。

### 步骤

1. **全景探索** — 先读整个仓库/workspace 目录结构，建立全局认知，再动手
2. **收集所有文档来源** — 飞书 wiki / 云文档 / 本地 Word/PDF / 代码中的 README
3. **统一转换为 Markdown** — 按类别归档到 `docs/` 目录（业务需求 / 设计文档 / 接口文档）
4. **完整性校验** — 逐篇检查内容是否完整、表格格式是否可识别、多来源做冲突检查

### 关键动作
```
全景探索 → 收集来源 → 批量转 MD → 按类别归档 → 完整性校验 → 冲突合并
```

### 经验教训
- 飞书 wiki 页面动态渲染，直接抓取可能内容不全，优先用 CLI 工具或导出功能
- 先建目录清单，再逐篇补正文，比一次性全量导入更稳健
- 文档多时用工具批量抓取正文，别手抄

---

## 阶段二：现有系统分析与复用评估

### 目标
评估已有系统哪些能力可以复用，哪些需要新建。

### 步骤

1. **分析现有系统** — 梳理 API 端点、数据模型、鉴权机制
2. **输出复用方案** — 哪些可复用 / 哪些需适配层 / 哪些必须新建

### 关键动作
```
系统分析 → 能力盘点 → 复用/新建决策
```

---

## 阶段三：需求文档与设计

### 目标
产出清晰的需求说明和设计方案。

### 步骤

1. **需求文档** — 按模块拆分，定义功能边界、用户角色、操作流程
2. **接口契约** — 定义数据格式和调用方式
3. **设计方案** — 整体架构、模块依赖、权限设计

### 关键动作
```
需求编写 → 接口契约 → 设计方案
```

---

## 阶段四：实现与验证

### 目标
按方案落地并持续验证。

### 步骤

1. **按步骤实现** — 一步步来，每步做完验证一步
2. **本地验证** — 尽量在本地跑通，缩短反馈环
3. **全盘盘点** — 定期全量检查遗漏和问题

### 关键动作
```
逐步实现 → 本地验证 → 全盘盘点 → 循环迭代
```

## 核心原则清单

| # | 原则 | 说明 |
|---|------|------|
| 1 | 文档先行 | 先把需求和设计整理清楚，再动手 |
| 2 | 复用优先 | 先看已有系统能给什么，再决定自己做什么 |
| 3 | 迭代验证 | 每做一步就验证一步 |
| 4 | 全盘盘点 | 定期全局检查，发现遗漏 |
| 5 | 一步步来 | 明确步骤，按步骤执行，不跳步 |
