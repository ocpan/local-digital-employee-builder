# AI Agent 产品预研流程

> 这份 skill 从 一个真实 Agent 项目的真实演进中抽象而来，重点不是“讲一套 AI 黑话”，而是把一个 Agent 产品从模糊想法收敛到可验证方案。

## When To Use

- 需要为一个新 AI Agent / 数字员工 / 智能助手做前期预研
- 需要判断它到底应该做成单技能助手、场景编排 Agent，还是更重的领域 Agent
- 需要在正式开发前先做一个可演示 Demo 或可验证 MVP
- 需要基于本地项目目录、旧对话记录、需求文档、demo、代码、截图提炼出一套产品路线
- 需要给出“为什么这么做”的技术路线选择，而不是只罗列工具名
- **要帮某人/某部门搭"数字员工/数字分身"飞书机器人做预研**（走下方「数字员工快速预研通道」，与 [[digital-twin-builder]] 配套：本 skill 定方案，那个 skill 落地）

## 不适用场景

- 只是要修一个现有代码 bug
- 只是要补一份 PRD，不需要做路线判断
- 只是要写大模型 Prompt，不涉及产品边界、工具链、流程、交互和 Demo

## 核心铁律

1. **先问题，后技术**：先说清用户、任务、场景、动作、风险，再谈模型、RAG、工作流。
2. **先任务拆解，后 Agent 化**：不是所有问题都该做成 Agent。先判断任务是否适合自动化、是否需要多轮、是否需要外部工具。
3. **先证据，后结论**：优先使用项目里的文档、代码、Demo、旧会话和反馈截图；没有证据的地方写“假设”或“待验证”，不要伪装成事实。
4. **先 Demo，后工程化**：预研阶段的核心目标是证明路径成立，而不是一上来就堆复杂架构。
5. **单人格入口优先**：如果用户群和任务链高度重叠，优先考虑“一个数字员工 + 内部分层路由”，不要轻易拆成多个外露机器人。
6. **写操作必须有确认点**：凡是创建工作项、写入系统、发消息、提交表单，必须定义明确确认点和失败回退。
7. **Prompt 不是状态机**：只靠上下文记状态非常脆。涉及长文档、多轮确认、跨系统动作时，要明确会话状态存储方案。
8. **没有数据源时不允许编造**：如果查询类能力还没接入真实数据源，统一返回“功能未上线/暂无权限/待补充”，而不是让模型猜。

## Required Inputs

- 产品方向或一句话问题定义
- 至少一个明确场景或目标用户群
- 本地项目目录、文档路径、代码路径、Demo 路径中的任意组合
- 如果用户要求“结合历史会话”，需要提供当前 workspace，或至少能定位到相关 transcript / session
- 可选：目标渠道（Web、飞书、企业微信、CLI、移动端）
- 可选：必须对接的系统或数据源

## 预研输出物

完成后至少给出以下结果：

1. 问题定义与边界
2. 证据清单与事实/假设分层
3. Agent 化判断结果
4. 技术路线选择结论
5. Demo / MVP 方案
6. 核心能力矩阵与优先级
7. 风险清单与验证计划
8. 下一步动作列表

## Workflow

### 1. 锁定预研问题，不要直接进入技术选型

先写出一条可判断真假的问题定义，格式建议：

```text
针对 <谁> 在 <什么场景> 下的 <什么任务>，是否应该用 <什么形态的 Agent 能力> 来提升 <什么指标>，并在 <什么渠道> 内完成验证？
```

同时明确：

- 主要用户是谁
- 主要任务是什么
- 成功指标是什么
- 当前不做什么
- 哪些动作属于高风险动作

如果这一层写不清，后面所有“模型选型”都没有意义。

### 2. 先做证据盘点，再做观点归纳

优先从本地项目材料提取证据，来源顺序建议如下：

1. 本地需求 / 方案 / 设计文档
2. 本地 Demo / HTML / 原型 / 海报
3. 当前项目代码入口、路由、状态管理、知识库结构
4. 历史会话记录
5. 用户反馈截图、问题清单、验收口径

强制动作：

- 列一份证据表，至少包含：来源、用途、事实、限制、待验证点
- 把信息分成三类：`事实`、`假设`、`待确认`
- 文档里出现的能力、代码里实现的能力、Demo 里展示的能力要分开记录，不能混为一谈

如果用户要求结合历史会话：

1. 先用 `session_store_sql` 查 session 元数据
2. 如果 turns / search index 没有正文，不要死磕，直接回退到 VS Code transcript `.jsonl`
3. transcript 的重点是抽“过程模式”：为什么这样做、如何迭代、哪里卡住，而不是逐句抄写

### 3. 做一次 Agent 化适配判断

把用户想做的事情拆成一组任务，然后逐项判断：

- 这是查询类、填单类、生成类、执行类，还是协同类任务
- 输入信息是否天然完整
- 是否需要多轮追问
- 是否需要读取知识库
- 是否需要调用外部系统
- 出错成本高不高

然后把任务分成四类：

1. **不该做成 Agent**：规则固定、表单明确、直接做页面或脚本更好
2. **L1 Skill**：单一目标、单条主路径、可控输入输出
3. **L2 Scenario Agent**：需要串联多个 Skill 才有价值
4. **L3/L4 候选**：只有在领域边界清晰、风险受控、确认点成熟时再考虑

判断原则见 [技术路线选择矩阵](./references/route-selection-matrix.md)。

### 4. 决定产品壳层和交互策略

这一层解决“用户看到什么”。

要判断：

- 是一个人格统一的入口，还是多个机器人并列
- 是只做 Web，还是 Web + 飞书双入口
- 是直接对话，还是“对话 + 卡片 + 快捷按钮”混合交互
- 是否需要双通道：`Fast Lane` / `Full Lane`
- 是否需要双语

推荐判断：

- **用户群高度重叠、任务链会串联**：优先“单人格 + 内部分层路由”
- **轻量需求和复杂需求并存**：优先“双通道”，让用户自己走快路或细路
- **企业内协作场景**：优先 Web + 飞书，而不是一上来做桌面客户端
- **只是内部概念验证**：静态 Web Demo 通常优于重前端工程

### 5. 决定技术路线，不要把“用大模型”当结论

至少要回答以下问题：

1. 是聊天壳层，还是工作流驱动，还是带工具编排的 Agent
2. 是否需要知识库；需要的话，是关键词检索、轻量查找，还是 RAG
3. 是否需要会话状态存储
4. 哪些动作只能建议，哪些动作可以执行
5. 是否需要权限校验、数据源接入和 no-hallucination 保护

通用建议：

- **表单收集 / 需求提报类**：优先 `对话 + 状态机 + 表单摘要`，而不是先上 RAG
- **知识问答类**：先验证知识源是否稳定，再决定是否上 RAG
- **跨系统执行类**：优先工作流和显式确认，不要完全放给开放式 Agent
- **长文档输入类**：必须加摘要和 token 预算策略，避免模型在确认环节丢状态
- **未接入真实数据源的查询类**：明确降级，不允许硬答

### 6. 设计 Demo，不要试图一次交付完整系统

预研阶段的 Demo 目标是“证明关键路径成立”。

Demo 类型优先级建议：

1. **单文件 HTML Demo**：最快，适合验证对话流、按钮流、卡片流、视觉口径
2. **本地 CLI / Skeleton**：适合验证路由、状态管理、知识库结构
3. **飞书单页海报 / 说明页**：适合对外讲能力边界和使用路径
4. **飞书机器人接入**：只有在渠道验证必须发生时才做
5. **全栈原型**：只在主路径已被验证后再进入

一个合格的 Demo 至少要覆盖：

- 1 条成功主路径
- 1 条复杂路径
- 1 条模式切换
- 1 条失败或拒答路径
- 1 条成功产物展示

### 7. 固化预研结论，必须落成结构化文档

不要只给用户一段口头分析。至少整理成以下结构：

1. 产品目标和边界
2. 用户场景与任务分解
3. 能力矩阵与优先级
4. 交互策略
5. 技术路线结论
6. Demo 方案
7. 风险与待验证项
8. 下一步落地计划

如果已有项目目录，建议把这些产物直接写回项目内 `docs/`、`.github/skills/` 或 `knowledge_base/`。

### 8. 用真实场景回放验证预研结论

最终至少做以下检查：

1. 用 3 到 5 条真实用户表达回放是否能正确路由
2. 检查轻量需求是否真的能快速走完
3. 检查复杂需求是否有足够结构化信息承接
4. 检查长文档输入是否会导致状态丢失
5. 检查查询类能力是否存在幻觉风险
6. 检查所有写操作是否有确认点和失败回退

## 数字员工快速预研通道（帮同事/部门搭飞书数字分身）

当预研对象明确是"给某人或某部门搭一个飞书数字员工/分身"时，走这条收敛通道——因为落地路径已被“个人分身”和“部门数字员工”两类真实项目验证过，预研不必从零发散，而是**对着一份成熟能力清单做取舍**。产出直接喂给 [[digital-twin-builder]] 落地。

### 四道准入闸（先判断值不值得搭，任一不满足先补）
1. **有稳定知识源吗**：有可沉淀成检索仓库的文档/纪要/规范 → 能做 RAG 问答；没有 → 只能纯工具型，价值窄。
2. **高频任务能自动化吗**：用下方 Agent 化判断过一遍；全是要人拍板的判断题 → 不适合。
3. **要连的业务系统有 CLI/API 吗**：查系统数据需要可编程入口（如 业务系统CLI）；纯 UI 系统只能靠人。
4. **飞书应用能发布吗**：企业自建应用要过管理员审核 + 配可用范围 + 开消息权限；组织卡这步就上不了线。

### 能力分层菜单（对着选，别一次全上）
- **L0 必做地基**：五层架构 + 知识库 RAG（命中引用来源、未命中降级不编造）+ 人格协议 PROTOCOL + 身份分权（本人全权/访客只读）。
- **L1 提速稳定**（上线即做）：runner 侧知识库预检索、系统提示瘦身、大脑后端抖动自动重试。
- **L2 业务系统接入**（要查数据才做）：标记协议 `[[X查询|...]]` 只读通道 + 白名单默认拒绝闸门 + SSO 授权自愈（绝不向用户要凭证）。
- **L3 主动性**（要"替他干活"才做）：后台长任务执行器（突破单轮超时的地基）→ 定时晨报/周报 → 写作产出流水线。
- **L4 对外服务**（要给访客/同事用才做）：DM 代答/私聊代理、观测（/stats + 心跳自愈）、管理命令。

每层的实现配方、反模式与血泪坑见 [[digital-twin-builder]] 的 `references/advanced-capabilities.md`。**预研阶段只需判断"这个对象需要到哪一层"**，不要在预研里写实现。

### 数字员工特有的预研输出（在通用 8 项之外补充）
- **准入闸结论**：四道闸逐条过，不满足的列为前置任务。
- **能力分层选型**：这个对象需要到 L 几，每层要不要、为什么。
- **访客策略**：是否对本人以外开放；开放的话访客能看什么、绝不能看什么（隐私/来源/私库）。
- **推广视角**（如果是"帮一批人搭"）：哪些是一次性底座、哪些每人要独立实例化（app/服务/知识库/凭证），见 digital-twin-builder 的 scaling-playbook。

### 反模式（数字员工专属）
- 无脑套现有分身的 config：别人的高频任务/业务系统/访客策略可能完全不同，必须独立预研。
- 知识库没就绪就搭对话：没数据源的问答分身只会编。
- 一次把所有进阶能力全上：按对方痛点排 L1→L3，别镀金。
- 把"用大模型"当结论：数字员工的价值在"接了谁的知识 + 连了什么系统 + 替他做了什么"，不在模型本身；飞书已有原生"私聊代理/数字分身"，自建版必须在"接本人知识库+业务系统"上有差异化，否则别自建。

## 真实项目 提炼出的可复用模式

从 一个真实 Agent 项目中，优先复用以下模式：

1. **知识库先行**：先把人格、设计方案、Demo 说明沉淀成可检索材料，再搭代码骨架
2. **单人格统一对外**：用户只看见“统一的对外人格名（如 Nova）”，内部再做多技能路由
3. **L1 先落地**：先把 `RD-01 需求提报` 跑通，再扩展 L2/L3
4. **轻重双通道**：Fast Lane 解决“快提交”， Full Lane 解决“要梳理”
5. **状态持久化**：确认类流程不能只依赖 prompt 上下文
6. **反幻觉护栏**：未接入数据时必须明确说“功能未上线/暂无权限”
7. **Web + 飞书双壳层优先**：企业协作产品先证明这两端，桌面端后置

证据整理模板见 [方法论来源与证据模板](./references/methodology-provenance.md)。

## Important Notes

- 预研不是“先定模型，再找场景”，顺序反了通常会失败
- 不要为了“像 Agent”而把本来适合表单/工作流的任务强行做成开放式对话
- Demo 阶段优先追求路径清晰和反馈真实，不追求完整后端
- 如果用户要求“结合之前会话”，要显式说明你用了哪些证据源，而不是只说“我综合判断”
- 任何路线建议都要附带取舍理由、适用前提和不选另一条路的原因
