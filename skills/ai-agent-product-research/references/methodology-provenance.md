# 方法论来源与证据模板

> 这份 skill 的方法论不是凭空来的，而是从一个真实 Agent 项目的迭代中提炼的。
> 这里给出**可复用的结构性结论**（谁做都成立），以及一份"把你自己项目的证据填进来"的模板——
> 预研时你应该用自己项目的真实材料替换下面的占位，让结论有据可依，而不是照抄。

## 一、从真实项目提炼的结构性结论（直接可复用）

### 结论 1：先建知识库，再起项目骨架
可复用动作顺序：
1. 先把人格、方案、边界固化成**知识材料**（可检索仓库）；
2. 再决定代码结构，而不是相反。
> 反面教训：先写业务代码、知识散落在脑子里，后期人格和边界很难收敛。

### 结论 2：先做 Agent 骨架，再按真实反馈修正产品边界
Agent 产品很少一次成型，通常在真实反馈后才补齐：权限边界、查询能力降级、确认点。
> 可复用：MVP 先跑主路径，把"边界/降级/确认"当作第二轮迭代项，别指望一版定稿。

### 结论 3：单人格统一对外，内部再做多技能路由
用户只应看见一个入口（一个数字员工），复杂度收在内部路由里，不要一上来拆成多个外露机器人。

### 结论 4：反幻觉是产品规则不是模型能力
查询类能力未接入真实数据源前，统一返回"功能未上线/暂无权限/待补充"，绝不让模型猜。

### 结论 5：状态要持久化，别只靠 prompt 上下文
涉及长文档、多轮确认、跨系统动作时，明确会话状态存储方案。

## 二、证据整理模板（用你自己的项目填）

预研时，对着你要搭的对象，把下表填满——有据可依的结论才可信：

| 证据来源 | 用途 | 事实（确认为真） | 限制/前提 | 待验证点 |
|---|---|---|---|---|
| 需求/方案/设计文档 | 定问题边界 | | | |
| 已有 Demo/原型 | 验证交互路径 | | | |
| 现有系统代码/接口 | 判断能连什么 | | | |
| 历史会话/沟通记录 | 抽演进模式 | | | |
| 用户反馈/问题清单 | 定痛点优先级 | | | |

填写铁律：
- 把信息分三类标注：`事实` / `假设` / `待确认`，不要把假设伪装成事实。
- 文档里写的能力、代码里实现的能力、Demo 里演示的能力**分开记**，不能混为一谈。
- 结合历史会话时，重点抽"过程模式"（为什么这样做、如何迭代、卡在哪），不是逐句抄。
