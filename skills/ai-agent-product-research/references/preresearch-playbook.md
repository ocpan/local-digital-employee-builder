# AI Agent 预研执行手册

## 目标

把一个 AI Agent 想法压缩成一条可验证的产品路线，而不是做成“看起来很智能但没有边界”的空壳。

## 固定执行顺序

### 1. 明确预研问题

先产出以下 5 项：

1. 目标用户
2. 高频任务
3. 使用渠道
4. 成功指标
5. 明确不做的范围

如果这些没写清，就不要开始“选模型”“选框架”。

### 2. 证据盘点

把输入材料按以下类别列出来：

| 类别 | 典型材料 | 预研用途 |
|---|---|---|
| 业务材料 | 需求文档、问题清单、立项材料 | 识别真实问题和目标 |
| 设计材料 | 方案文档、流程图、交互稿、海报 | 明确对外口径和交互方式 |
| Demo 材料 | HTML、截图、原型、录屏 | 反推出主路径和展示重点 |
| 代码材料 | 入口文件、路由、状态管理、知识库代码 | 判断已有实现基础 |
| 历史会话 | transcript、session 元数据 | 提取演进过程与卡点 |
| 用户反馈 | 飞书截图、bug 列表、评审意见 | 识别真实风险和体验问题 |

输出要求：

- 每类至少写清：事实、限制、待验证点
- 没有事实支撑的地方显式标注为“假设”

### 3. 任务拆解

把用户要完成的事情拆成最小任务单元，每个单元至少写：

- 输入是什么
- 输出是什么
- 是否需要追问
- 是否需要知识库
- 是否需要外部系统写入
- 错了会造成什么后果

这一步是后面技术路线选择的基础。

### 4. 做 Agent 化判定

常用判定逻辑：

| 任务类型 | 建议形态 |
|---|---|
| 信息收集明确、字段固定 | 表单 / 工作流优先 |
| 单一任务、需要自然语言入口 | L1 Skill |
| 多技能串联才有价值 | L2 Scenario Agent |
| 需要长期自治和领域闭环 | L3/L4 候选 |
| 只是展示或查找 | 普通页面 / 搜索优先 |

结论必须说明“为什么不是另一种形态”。

### 5. 决定产品壳层

先决定用户层看到什么：

- 一个数字员工还是多个机器人
- Web、飞书、两者兼有，还是别的渠道
- 纯对话还是对话 + 卡片 + 按钮
- 是否需要轻重双通道

经验判断：

- 同一批用户会连续做“提需求、查进度、问配置”这类事，优先单人格入口
- 企业协作场景优先飞书 / Web
- 如果 80% 需求都很轻，优先给一条极速路径

### 6. 决定技术路线

至少回答这些问题：

1. 是否需要大模型
2. 是否需要知识库 / RAG
3. 是否需要状态存储
4. 是否需要工具调用
5. 哪些写操作必须确认
6. 是否存在权限和数据源问题

强制规则：

- 长文档输入必须考虑摘要和 token 预算
- 查询类能力没有真实数据源就不要开放
- 多轮确认类流程必须设计状态持久化

### 7. 做 Demo 策略

预研期 Demo 的目标是验证，不是交付。

推荐顺序：

1. 静态或半交互 HTML Demo
2. 本地 Skeleton / CLI
3. 飞书单页或机器人接入
4. 再决定是否进入全栈实现

Demo 至少要展示：

- 主路径
- 复杂路径
- 失败回退
- 成功产物

### 8. 沉淀预研结果

至少输出以下内容：

- 产品定位
- 用户场景
- 核心能力
- 技术路线
- Demo 方案
- 风险清单
- 下一步计划

## 常见失败模式

1. 一上来讨论模型和 RAG，没有先说清任务
2. 用开放式对话去承载本质上是表单的流程
3. 只做 Full Lane，没有给轻量场景留快路
4. 把 prompt 当成持久状态，长上下文后流程断裂
5. 没接数据源就开放查询功能，导致幻觉
6. Demo 做得很重，但没有验证核心路径

## 某 Agent 项目（示例） 对照案例

### 某 Agent 项目（示例） 做对的地方

- 先有知识库和人格定义，再搭骨架
- 统一为示例数字员工A / 示例数字员工A 单人格入口
- 把能力先压到 L1，再扩展到 L2/L3
- 用 Fast Lane / Full Lane 处理轻重需求差异
- 在反馈后补齐了 no-hallucination、状态持久化、权限边界

### 某 Agent 项目（示例） 暴露出的风险

- 飞书长文档读入后触发 token 压力，导致确认态丢失
- 未接真实数据时容易被用户误以为能查询项目详情
- 查询、提交、工时等能力边界需要更清晰的可见性控制
