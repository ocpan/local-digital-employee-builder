# AI Agent 预研工具推荐矩阵

## 1. 本地材料定位与阅读

| 目标 | 推荐工具 | 何时使用 | 不建议做法 |
|---|---|---|---|
| 快速盘点项目目录 | `search_subagent` / `Explore` / `list_dir` | 刚接手项目、要找 docs / demo / code 入口时 | 一开始就全量 grep 整个仓库 |
| 定位特定关键字 | `grep_search` | 已经知道关键词、要找精确文本时 | 为同义词分别搜很多轮 |
| 读取核心文件 | `read_file` | 已经锁定文件后，需要读正文时 | 过度碎片化小段读取 |

## 2. 历史会话回溯

| 目标 | 推荐工具 | 何时使用 | 不建议做法 |
|---|---|---|---|
| 查历史 session 元数据 | `session_store_sql` | 想先知道有没有相关会话时 | 直接假设本地 session 有完整正文 |
| 查 transcript 正文 | `read_file` + transcript `.jsonl` | `turns` / `search_index` 空或无正文时 | 在 session_store 里反复无结果重试 |
| 批量筛 transcript | `run_in_terminal` + `rg` | transcript 文件很多，需要先粗筛时 | 把所有 transcript 全量输出到对话里 |

## 3. 结构化产物沉淀

| 目标 | 推荐工具 | 何时使用 | 不建议做法 |
|---|---|---|---|
| 沉淀 skill / reference | `apply_patch` | 新建或更新 `.github/skills/...` 文档时 | 用 shell 重定向零散写多个文件 |
| 保存方法论或经验 | repo memory | 形成可复用经验后 | 只留在当前对话里，不做沉淀 |

## 4. Demo 原型搭建

| 目标 | 推荐工具 / 技术 | 何时使用 | 不建议做法 |
|---|---|---|---|
| 验证对话主路径 | 单文件 HTML + CSS + JS | 要快速验证入口、卡片、按钮、成功态时 | 预研阶段就上重前端框架 |
| 验证路由 / 状态 | Python / Node CLI skeleton | 先验证人格、路由、状态机时 | 一开始就做完整服务端 |
| 本地预览 demo | `npx serve` / `python -m http.server` | 静态文件需要本地访问时 | 上复杂部署流水线 |
| 对外分享静态 demo | `lark-apps` | 需要公网可访问链接时 | 手动打包到不稳定环境 |

## 5. 飞书场景推荐

| 目标 | 推荐工具 | 何时使用 |
|---|---|---|
| 读写云文档 | `lark-doc` | 需要读取 PRD、生成说明文档、更新云文档时 |
| 管理文件和目录 | `lark-drive` | 需要上传 demo、整理资料、复制文件时 |
| 发送消息 / 群内验证 | `lark-im` | 要验证机器人话术或发通知时 |

## 6. 技术选型建议

| 问题类型 | 推荐起步方案 |
|---|---|
| 需求提报 / 表单收集 | 对话入口 + 状态机 + 结构化摘要 |
| 知识问答 | 检索优先；知识稳定后再上 RAG |
| 跨系统执行 | 工作流优先，关键节点确认 |
| 复杂协同 | 单人格 + L1/L2 路由 |
| 仅做概念验证 | 静态 Demo + 假数据成功态 |

## 7. 真实项目 验证出来的经验

1. `session_store_sql` 适合找会话壳信息，不一定能拿到 turn 正文。
2. transcript `.jsonl` 才是回收真实旧对话过程的可靠兜底。
3. HTML Demo 很适合验证 Fast Lane / Full Lane 这类交互分流。
4. 只靠 Prompt 维护确认态会在长文档场景里失效，必须补状态存储。
