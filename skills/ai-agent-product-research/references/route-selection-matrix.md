# AI Agent 技术路线选择矩阵

## 1. 产品形态选择

| 判断问题 | 推荐路线 | 适用条件 | 不选另一条的原因 |
|---|---|---|---|
| 用户群和任务链高度重叠吗？ | 单人格入口 + 内部分层路由 | 同一批用户会连续提需求、查进度、问配置 | 多机器人会让入口分散、心智变重 |
| 用户群完全不同吗？ | 多入口 / 多机器人 | 场景完全不同、互不串联 | 单人格会变成“大而全” |

## 2. 执行架构选择

| 场景 | 推荐架构 | 说明 |
|---|---|---|
| 单一目标、单条主路径 | L1 Skill | 例如需求提报、工时填写 |
| 多技能串联才有价值 | L2 Scenario Agent | 例如“提需求 + 提交 + 查流转” |
| 需要领域级自治 | L3 候选 | 只有边界、权限、确认点成熟后才进入 |
| 强治理下的业务闭环 | L4 候选 | 预研阶段通常不建议直接做 |

## 3. 交互模式选择

| 判断问题 | 推荐路线 |
|---|---|
| 80% 请求都很轻吗？ | 加 `Fast Lane` |
| 复杂需求需要结构化梳理吗？ | 加 `Full Lane` |
| 轻重需求并存吗？ | 双通道并存 |
| 所有请求都高度标准化吗？ | 直接表单或工作流，不一定需要双通道 |

### 某 Agent 项目（示例） 路线

- `Fast Lane`：3 个字段，30 秒提交，不生成 PRD
- `Full Lane`：结构化梳理，生成 PRD / 文档

## 4. 知识层选择

| 场景 | 推荐路线 | 原因 |
|---|---|---|
| 主要是表单收集 / 提交 | 不急着上 RAG | 问题在流程和状态，不在检索 |
| 需要回答稳定知识 | 轻量检索或 RAG | 必须依赖文档内容 |
| 知识源仍频繁变动 | 先做可控检索 | 过早上 RAG 容易维护困难 |

## 5. 状态管理选择

| 场景 | 推荐路线 | 原因 |
|---|---|---|
| 单轮问答 | Prompt / 无状态即可 | 复杂度低 |
| 多轮确认、长文档输入 | 会话状态存储 | 防止 token 导致状态丢失 |
| 涉及待提交草稿 | 状态存储 + 明确阶段标记 | 用户可能修改、回退、确认 |

### 某 Agent 项目（示例） 路线

- 需求提报类流程必须有状态存储
- “确认”不能只靠模型记忆上下文

## 6. 数据与权限选择

| 判断问题 | 推荐路线 |
|---|---|
| 有真实数据源吗？ | 有则接入；没有则明确降级 |
| 有权限校验吗？ | 有写和查的动作都要定义权限策略 |
| 高风险写操作多吗？ | 先工作流化，再 Agent 化 |

### 强制规则

1. 没有数据源时不要开放查询详情能力。
2. 没有权限校验时不要开放敏感数据查询。
3. 所有外部写入都必须有明确确认点。

## 7. 渠道壳层选择

| 场景 | 推荐路线 | 说明 |
|---|---|---|
| 企业协作内场景 | Web + 飞书优先 | 低门槛，贴近真实使用场景 |
| 只是做概念验证 | Web 单页即可 | 快速、稳定、易演示 |
| 桌面端强依赖吗？ | 只有明确要求才做 | 预研阶段通常后置 |

### 某 Agent 项目（示例） 路线

- 优先 Web 工作台 + 飞书机器人
- 明确“不做桌面客户端”

## 8. Demo 形态选择

| 预研目标 | 推荐 Demo |
|---|---|
| 验证对话与交互路径 | 静态 / 半交互 HTML Demo |
| 验证路由 / 状态 / 数据结构 | 本地 skeleton |
| 验证渠道可达性 | 飞书机器人接入 |
| 验证对外宣传口径 | 单页海报 / landing page |

## 9. 推荐默认路线

如果用户还没有明确限制，默认按下面路线起步：

1. 单人格入口
2. Web + 飞书双壳层
3. L1 Skill 先落地
4. 轻重双通道
5. 状态存储而非纯 Prompt
6. 没有真实知识和数据时，不先上复杂 RAG
7. 用 Demo 先证伪，再决定工程化范围
