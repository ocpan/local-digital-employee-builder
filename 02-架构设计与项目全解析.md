## 一、四条设计哲学（第一性原理）

这套方案不是「又一个聊天机器人脚手架」，它的每个决策都服务于四条第一性原理：

| # | 原则 | 一句话解释 | 带来的能力 |
|---|------|-----------|-----------|
| 1 | **分层解耦** | 感知 / 大脑 / 知识 / 行动 / 状态，五层各司其职，接口清晰 | 任何一层可单独替换、升级、测试 |
| 2 | **本地优先（Local-First）** | 代码、凭证、知识、状态全部在你自己的机器上 | 数据可控、透明可审计、无供应商锁定 |
| 3 | **工具无关 & 模型无关** | 搭建端换任意 AI 编码工具，大脑端换任意兼容 API | 不被任何单一厂商绑架 |
| 4 | **自我演进（Self-Evolving）** | 机器人能读写自己的代码/知识/配置，按指令长出新能力 | 用自然语言就能给它做「模块开发」 |

贯穿这四条的，是一条隔离铁律——**「不给权力，给协议」**：
大脑（LLM）被关在沙箱里，它不能直接碰外部世界；所有对外动作都写成结构化"标记"，
由沙箱外的 `runner` 校验后代跑。这让一个足够强的大脑，也无法越过工程设定的权限边界。

## 二、项目结构逐文件解析

先看全貌，再逐个说明每个文件/目录「是什么、干什么、你要不要动它」。

```
数字员工快速搭建包/
├── 00-先看这里-START-HERE.md      快速上手指南（怎么跑起来）
├── 01-申请飞书机器人指引.md        申请飞书机器人的图文步骤
├── 02-架构设计与项目全解析.md      本文档（为什么这样设计）
│
├── skills/                        方法论技能库（AI 工具搭建时会读）
│   ├── digital-twin-builder/       ★核心：怎么搭 + 进阶能力配方 + 规模化
│   ├── ai-agent-product-research/  预研：该不该搭、搭到哪一层
│   └── product-methodology/        产品方法论底座（从零做产品的通法）
│
└── bot/                           机器人本体（真正跑起来的进程）
    ├── runner.py                   主程序：收消息 → 大脑 → 检索 → 回复
    ├── config.json                 总配置：填你的信息（唯一必改文件）
    ├── prompts/
    │   └── PROTOCOL.md             机器人的人格与铁律（运行时注入大脑）
    ├── brain_home/                 大脑配置目录（可换任意 OpenAI 兼容 API）
    │   ├── config.toml             大脑后端：网关地址、模型、推理参数
    │   └── auth.json               大脑密钥（本地保存，.gitignore 兜底）
    ├── knowledge_base/             知识库（机器人的"记忆"）
    │   ├── docs/                    原始层：你的 .md 文档（权威来源）
    │   ├── wiki/                    编译层：按 schema 生成的实体/概念页
    │   ├── rag/                     索引层：检索用的 chunks + index（脚本生成）
    │   └── scripts/
    │       ├── build_rag_index.py   构建/重建检索索引
    │       └── rag_search.py        关键词检索（大脑靠它找答案）
    ├── state/                      运行状态（本地 JSON，重启可恢复）
    │   ├── processed_ids.json       已处理消息（防重复回复）
    │   ├── sessions.json            会话映射（续聊不断档）
    │   └── ...                      后台任务、定时任务等状态
    ├── logs/                       运行日志
    ├── deploy/
    │   ├── start.sh                 前台启动（含断线自动重启）
    │   └── com.mybot.template.plist 7×24 常驻模板（macOS launchd）
    └── .gitignore                  保护凭证/状态/日志不进版本库
```

### 逐项作用说明

| 路径 | 作用 | 你要不要改 |
|------|------|-----------|
| `bot/runner.py` | **中枢守护进程**。消费飞书事件流、过滤去重、鉴权、调大脑、检索知识、回卡片、管会话与状态。它是"跑得起来的最小核心"，一份逻辑驱动所有分身。 | 一般不改；加高级能力时由 AI 帮你改 |
| `bot/config.json` | **唯一必改文件**。机器人名、你的飞书 app_id、你的 open_id、访问模式、大脑后端选择（codex/claude）、知识库路径、会话 TTL 等全在这里。 | ✅ 必改（填你的信息） |
| `bot/prompts/PROTOCOL.md` | **人格与铁律说明书**，每轮作为系统提示注入大脑。定义身份、主人是谁、访客/本人分权、知识来源约束、回答流程、硬性禁令。 | ✅ 建议按你的场景改写 |
| `bot/brain_home/config.toml` | **大脑后端配置**：接哪个 OpenAI 兼容网关、用什么模型、推理强度。换 API 就改这里的 `base_url` 和 `model`。 | 换大脑服务时改 |
| `bot/brain_home/auth.json` | **大脑密钥**（API Key）。本地保存，被 `.gitignore` 排除，不进版本库。 | ✅ 填你的 Key |
| `bot/knowledge_base/docs/` | **原始知识层**：放你自己的 `.md` 文档，是机器人回答的权威来源。目录可分「公开 / 私有」控制访客可见范围。 | ✅ 换成你的文档 |
| `bot/knowledge_base/wiki/` | **编译知识层**：由脚本/大脑按 schema 生成的实体页、概念页，检索时加权。可选、可重建。 | 一般自动生成 |
| `bot/knowledge_base/rag/` | **检索索引层**：`build_rag_index.py` 生成的 chunks 与倒排索引，`rag_search.py` 靠它做关键词检索。 | 文档变了就重建 |
| `bot/knowledge_base/scripts/build_rag_index.py` | **构建索引**：遍历 docs/wiki，切块、抽取分面（part/product/module），产出可 GitHub 托管的轻量索引。 | 不改 |
| `bot/knowledge_base/scripts/rag_search.py` | **检索器**：流式扫描索引分片，支持分区过滤（访客强制限公开区）。 | 不改 |
| `bot/state/` | **运行状态**：全部本地 JSON 落盘。`processed_ids` 防重复回复、`sessions` 让重启后续聊不断档，还有后台/定时任务状态。**无 DB、无 Redis**。 | 不动（自动维护） |
| `bot/logs/` | 运行日志，排障看这里。 | 不动 |
| `bot/deploy/start.sh` | **前台启动脚本**：`cd` 到 bot 目录循环拉起 `runner.py`，崩了 5 秒重启（进程级兜底）。 | 直接用 |
| `bot/deploy/com.mybot.template.plist` | **7×24 常驻模板**（macOS launchd）：开机自启、常驻不掉、日志重定向。Linux 服务器换成 systemd 同理。 | 部署时用 |
| `bot/.gitignore` | 保护 `auth.json`、`state/*.json`、日志不进版本库。 | 不动 |
| `skills/digital-twin-builder/` | **核心方法论技能**：教 AI 工具怎么一步步帮你搭，含进阶能力配方（后台任务、定时晨报、授权自愈…）与规模化复制手册。 | 不改（AI 会读） |
| `skills/ai-agent-product-research/` | **预研技能**：帮你判断该不该搭、搭到哪一层、要连哪些系统。 | 不改 |
| `skills/product-methodology/` | **产品方法论底座**：从零做一个产品的通用打法。 | 不改 |

## 三、五层架构：为什么这样设计，好处是什么

机器人内部是一条清晰的五层流水线，一条消息自上而下流过：

```
飞书用户/群
   │
   ▼
L1 感知层  ── 听见：事件消费 · 过滤去重 · @识别 · 访问控制
   │
   ▼
L2 大脑层  ── 想清楚：headless LLM（沙箱内）· 人格每轮注入 · 会话 resume
   │
   ▼
L3 知识层  ── 有依据：docs 原始 → wiki 编译 → rag 索引，先检索后回答
   │
   ▼
L4 行动层  ── 干得动：飞书卡片 · 思考流式 · 标记协议网关（校验后代跑）
   │
   ▼
L5 状态层  ── 记得住：本地 JSON 落盘，重启可恢复
```

### 这样分层的三大好处

- **可扩展**：想加能力，就在对应层"插一块"，不动其它层。
  加附件分析 → 扩 L1；换更强的模型 → 换 L2；接新的业务系统 → 在 L4 加一个只读白名单命令；加定时晨报 → L5 加一个调度状态。**层与层之间只靠约定接口通信，改动被限制在局部。**
- **可替换**：每层都能整块换掉而不牵连全局。
  大脑从企业网关换成 Claude、检索从关键词换成向量、状态从 JSON 换成数据库——只要接口不变，其余层无感。
- **可自我迭代**：因为所有层都是**纯文件 + 纯命令**，机器人自己也能读懂、改写它们（见第七章）。

### 差异只收敛在三处 → 规模化成本极低

搭第二个、第三个数字员工时，`runner.py` 一行都不用改，你只需要换三样东西：

| 换什么 | 决定了 |
|--------|--------|
| `config.json` | 它是谁、连谁、对谁开放 |
| `PROTOCOL.md` | 它的人格、分权、话术 |
| `knowledge_base/` | 它懂什么 |

**「搭第 N 个分身 = 换三个文件」**——这是这套架构最值钱的性质。

## 四、工具无关：不只 Claude Code，任何 AI 工具都能用

文档里默认用 Claude Code 演示，但这只是"其中一种施工队长"。本方案在**两端**都不绑定厂商：

### 搭建端：任何 AI 编码工具都能驱动

`skills/` 是标准的 Markdown 方法论文件，`bot/` 是标准的 Python + JSON + Markdown。
任何能读文件、跑命令的 AI 编码工具都能照着 `skills/digital-twin-builder/SKILL.md` 帮你搭：

- **Claude Code**、**Cursor**、**Codex CLI**、**Windsurf**、**GitHub Copilot（Agent 模式）**、**通义灵码**…
- 甚至不用 AI，一个人照着 `00-先看这里` 手动搭也完全可行——只是慢一点。

> 换工具时把那句"请读 `skills/digital-twin-builder/SKILL.md`，按步骤带我搭"发给新工具即可，流程一致。

### 大脑端：任何 OpenAI 兼容 API 都能接

`config.json` 的 `brain.provider` 支持切换，`brain_home/` 是通用的大脑配置目录：

- `provider = "codex"`：经 Codex CLI 桥接**任意 OpenAI 兼容网关**——改 `brain_home/config.toml` 的 `base_url` 与 `brain_home/auth.json` 的 key，就能接企业网关、OpenAI、Azure、各家兼容服务。
- `provider = "claude"`：直接走 Claude API（Anthropic）。

**没有任何一处代码写死了"必须用某一家的模型"。** 换大脑，改配置即可，`runner.py` 不动。

## 五、本地优先（Local-First）的价值

这套方案把一切都放在**你自己的机器**上，而不是托管在某个 SaaS 平台。这带来四个别人拿不走的好处：

- **凭证依赖存于本地** 🔐
  飞书 token、大脑 API Key（`brain_home/auth.json`）、SSO 凭证全在本地文件里，被 `.gitignore` 兜底，不上传任何第三方平台。谁能碰你的凭证，由你决定。
- **知识库完全可控** 📚
  知识就是一堆 `.md` 文件躺在 `knowledge_base/docs/` 里。想加就丢文件、想删就删文件、想改就改文本，还能纳入你自己的私有 Git 做版本管理。**内容透明、来源可审计、增删即时生效**——不像 SaaS 那样把你的知识关进它的数据库。
- **迭代简单** ⚡
  改行为 = 改 `PROTOCOL.md`；改知识 = 改 `docs/` 重建索引；改能力 = 让 AI 改 `runner.py`。**没有"重新发布应用""等平台审核"这一步**，改完重启进程即可生效。
- **零外部运行时依赖** 🧩
  状态全是本地 JSON，**不需要数据库、不需要 Redis、不需要消息队列**。单机即可跑，重启靠会话 resume + 队列复位自动恢复。

## 六、部署自由：本地或任意服务器，零云端平台依赖

机器人本质上就是**一个 Python 进程 + 一条 lark-cli 长连接**，因此它能跑在几乎任何地方：

| 部署环境 | 常驻方式 |
|----------|---------|
| 你的 Mac | launchd（本包已带模板 `com.mybot.template.plist`）+ `caffeinate` 防休眠 |
| Linux 服务器 / 云主机 | systemd service（同样几行配置，让 AI 帮你生成）|
| 公司内网机 / 树莓派 / NAS | 前台 `start.sh` 或 nohup / tmux 常驻 |
| 容器 | 一个 `python3 runner.py` 的镜像即可 |

**关键点：它不依赖任何云端编排平台。**
不锁定 Dify / Coze / 扣子 / n8n 这类 SaaS——不需要在它们的画布上搭流程、不受它们的额度和可用性约束、不把数据交给它们。
唯一的外部依赖是：① 飞书开放平台（收发消息，官方能力）② 一个 LLM API（可自选、可自建）。
**换句话说，只要有一台能联网的机器，你就能拥有一个完全归属自己的数字员工。**

## 七、自我演进：让机器人自己开发新模块

这是本方案最接近"世界顶级"的地方——机器人不是一个死程序，而是一个**能改造自己的 Agent**。

### 为什么它能自我演进

因为它的"身体"（`runner.py`）、"人格"（`PROTOCOL.md`）、"记忆"（`knowledge_base/`）、"配置"（`config.json`）
**全部是它自己能读能写的纯文本文件**，而它的大脑又是一个具备读文件、写文件、跑命令、跑测试能力的编码级 LLM。
在**主人 full-access 沙箱**下，你可以直接对它下达开发指令：

> 「给你自己加一个『每天早上 9 点把昨天的提问汇总成晨报发我』的能力，写完自测通过再上线。」

它会：读自己的 `runner.py` → 设计调度模块 → 写代码 → 跑测试 → 汇报 → （确认后）重启自己生效。
**用自然语言，就完成了一次真正的模块开发。**

### 自我改造的安全铁律（防止"改崩自己"）

赋予自我修改能力的同时，`PROTOCOL.md` 内置四条铁律兜底（源于真实事故：机器人为加定时功能重启自己，却杀掉了正在回复的轮次）：

1. **仅主人明确确认后**，才能改自身代码/配置；
2. 改完必须**测试全绿**才生效，失败自动回滚并汇报；
3. **先发完工汇报，再重启**（不让重启吞掉未送达的消息）；
4. 重启**优先建议人工执行**，自动重启前先发"即将重启"提示。

> 这套"能自我进化，但有安全护栏"的设计，正是顶级 Agent 方案与普通脚本的分水岭。

## 八、演进路线与方案对比

### 从跑起来到长成数字员工

```
第 0 步：基础问答      → 知识库 + 分权，能替你答重复问题（1-2 小时搭完）
   ↓
第 1 步：进阶能力      → 附件分析 / 后台长任务 / 每日晨报 / 查业务系统
   ↓                     （读 skills/digital-twin-builder/advanced-capabilities.md）
第 2 步：自我迭代      → 用自然语言让它给自己开发新模块
   ↓
第 3 步：规模化复制    → 换 config/PROTOCOL/知识库三件套，复制给同事/部门
   ↓                     （读 skills/digital-twin-builder/references/scaling-playbook.md）
```

### 本方案 vs 云端编排平台（Dify / Coze 类）

| 维度 | 本方案（本地 Agent） | SaaS 编排平台 |
|------|---------------------|--------------|
| **数据与凭证** | 全在本地，你掌控 | 托管在平台侧 |
| **知识库** | 本地 .md，透明可审计、可 Git 版本化 | 平台数据库，黑盒 |
| **模型选择** | 任意 OpenAI 兼容 / Claude，可自建 | 受平台支持列表限制 |
| **搭建工具** | 任意 AI 编码工具，无绑定 | 平台专有画布 |
| **部署** | 本地 / 任意服务器 / 容器 | 只能跑在平台上 |
| **迭代方式** | 改文件即生效，可自我演进 | 平台画布拖拽，受能力边界约束 |
| **供应商锁定** | 无 | 强 |
| **适合谁** | 想要长期掌控、深度定制、可进化的数字员工 | 想要快速拖拽出一个轻量 Bot |

---

> **一句话总结**：这套方案的价值不在"能做一个聊天机器人"，而在于它是一个
> **本地掌控、工具无关、可无限复制、还能自我进化**的数字员工底座——
> 你搭的不是一个 Bot，而是一个属于你自己的、会成长的 AI 员工。
